0\. まず再定義するKPI（ここが設計の芯）

プロダクトKPI（最重要）



Time-to-Next-Action



起動から「次の行動（食事/運動/休養/受診検討）」を選ぶまでの時間



Recommendation Adoption Rate



AI提案が読まれた率ではなく、実行された率



Daily Check-in Completion



1日1回の確認フロー完了率（状態把握→提案確認→1つ実行）



Data Coverage for Advice



助言に必要な最低データが揃っている日率



7日継続率 / 30日継続率



健康アプリは継続が本体。UIの美しさは従属



やってはいけないKPI



表示カード数



連携データ種類数



グラフ表示回数



これらは「作ってる感」は出るが、ユーザー価値に直結しない。



1\. プロダクトのコア価値（1文で定義）



「複数の健康データを統合し、複数AIの視点を調停して、今日やるべき行動を1〜3個に絞って提示する」



これを外すと、単なる健康データビューアーになる。

3\. 画面設計（情報設計）※ホーム中心に全体整合

3-1. ホーム（主役）



ホームは「データ一覧」ではなく 1日1回のブリーフィング面。



ホームの構成（推奨）

ブロックA：今日の結論（最上段）



今日の状態要約（1行）



例:「回復は中程度、活動量不足、食事ログ不足で栄養提案の精度は低め」



確信度表示



高 / 中 / 低（理由つき）



データ鮮度



更新時刻、欠損あり表示



ブロックB：最優先アクション（核）



三者合意の最優先行動（1件）



領域別の次点（最大2件）



運動



栄養



回復/生活



各アクションに「なぜ今それか（根拠1〜2行）」



ブロックC：例外処理（欠損・異常・注意）



欠損：食事未入力など（責めない文言）



異常：普段より大きなズレ



注意：安全上の確認が必要なもの（受診勧奨含む）



ブロックD：根拠カード（少数）



睡眠 / 活動 / 体組成 / バイタル / 栄養 のうち



今日の結論に効いたものだけ表示



すべて表示しない（認知負荷が上がるだけ）



ブロックE：AI会話導線



状況に合わせた質問候補（3つ）



「睡眠が短い日の運動はどう調整？」



「体重増の原因候補を3つ」



「今日の昼食の選び方は？」



重要：あなたの不満（データが散らばって“使える知見”にならない）は、ホームをこの構造に変えるとかなり解消される。

逆に、カードを増やしても解消しない。



3-2. 深掘り画面（ドリルダウン）



ホームで結論を出し、深掘りは「検証用」に限定。



タブ構成（推奨）

栄養

バイタル

4\. AIエージェント設計（役割分担と調停が本体）



あなたの差別化はここ。

ただし「3人の専門家っぽく喋る」だけだと価値は出ない。調停ロジックが必要。



4-1. エージェント役割（推奨）

① トレーナーAI



活動量、睡眠、疲労指標を見て



運動の種類 / 強度 / 時間を提案



② 栄養士AI



食事ログ、体重推移、活動量から



次食 / 1日のPFC方向性 / タイミング提案



③ 医療安全AI（“医師AI”よりこの表現が安全）



バイタル・SpO2・血圧・症状メモから



診断しない



赤旗チェック（受診/相談推奨）



不確実性・限界を明示



「医師AI」と前面に出すと規制・期待値・法的リスクが一気に跳ねる。

MVPでは “医療安全ガードレール担当” に落とすのが現実的。



4-2. 調停（Orchestrator）設計

入力



正規化データ



データ品質スコア



ユーザー目標



過去の実行率（提案を守れるタイプか）



処理



各エージェントが候補提案を出す



競合検出（例：運動強度上げる vs 回復優先）



安全優先ルール適用



実行可能性スコアで並べ替え



最終提案1〜3件に圧縮



出力



最優先行動



代替案



根拠



確信度



注意点



4-3. 衝突時の優先順位（ルールを固定）



これを決めないと、毎回出力がブレる。



推奨優先順



安全（赤旗・急変可能性）



回復（睡眠不足・過負荷）



継続可能性（今日はできるか）



目標最適化（減量効率など）



理想論（最適だけど実行困難）



あなたの癖（筋の通った正解に寄りがち）を踏まえると、ここは特に重要。

アプリ側が「理論上の最適」より「今日の実行最適」を優先するよう、システムで矯正すべき。

10\. 失敗しやすいポイント（先に潰す）

失敗1：データ統合を目的化する



症状：連携数は増えるが価値が増えない



対策：ホームの「最優先行動」品質を最上位KPIに置く



失敗2：AIを人格化しすぎる



症状：3人の会話は面白いが、結論が遅い



対策：ユーザーには「調停済み結論」を先に出す



失敗3：食事入力を前提にしすぎる



症状：継続率が落ちる



対策：欠損でも回る設計＋必要時だけ補完依頼



失敗4：医療っぽい表現を先行



症状：規制/審査/期待値の地雷を踏む



対策：MVPは生活改善支援＋安全ガードに寄せる



失敗5：ホームに全部載せる



症状：結局、ただの集計画面



対策：ホームは結論面、詳細は深掘りへ分離
